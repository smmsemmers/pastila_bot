### Настройки-рычаги (что можно крутить)
- **Дефолт под задачу** (`TASK_RECOMMENDED`) — рекомендованная модель per task.
- **Выбор по чату** (`_MODEL_CHOICE`, хранится в Google Sheet вкладка `llm_config`) — перебивает дефолт. ⚠️ Сейчас есть баг: выбор иногда не переживает рестарт Render.
- **Авто-режим** — выбор по сложности запроса (дёшево→дорого).
- **/trim 0–10** — уровень обрезки контекста; режим `truncate` / `squeeze` (умное сжатие моделью SQUEEZE_MODEL).
- **env:** `VISION_MODEL`, `SQUEEZE_MODEL`, `OPENAI_MODEL` (Whisper), `OPENROUTER_API_KEY`.
- Параметры вызова: temperature, max_tokens, thinking_budget (для Opus).

## РАЗДЕЛ 3. ФИНАЛЬНЫЙ ПРОМТ (задание для исследования)

Ты — эксперт по выбору LLM под задачи и по UX Telegram-ботов. На основе Разделов 1–2 выше спроектируй оптимальную схему выбора моделей. Дай конкретный результат:

1. **Модель под каждую функцию.** Для каждой функции из таблицы (Раздел 1) порекомендуй лучшую модель из каталога (Раздел 2) с обоснованием: качество vs цена vs скорость. Учти: разбор файлов и стратегия — важное качество (флагман ок); голос-роутинг, детектор инсайтов — дёшево; чат — баланс ума и цены. Дай таблицу «функция → рекомендованная модель → почему → запасная».

2. **Дефолт для чата** + быстрая смена. Предложи дефолтную модель общения и механику МГНОВЕННОЙ смены прямо в чате (например Opus ↔ Fable 5 ↔ GPT-5.5) — UI-friendly, в 1 тап. Опиши инлайн-кнопки: набор кнопок под сообщением для быстрого переключения модели чата.

3. **Инлайн-кнопки для редких функций.** Для редко вызываемых функций (/strategy, /deep, /pin, /post, vision) — предложи компактный инлайн-UI выбора модели, чтобы не засорять чат. Как сгруппировать.

4. **Команда «проверить обновления модели под функцию».** Спроектируй команду, которая: (а) проверяет, не появилась ли новая/лучшая модель под конкретную функцию; (б) предлагает смену, если найдена лучше; (в) САМА выполняется на модели ЧАТА (default), а не на /analyze. Опиши логику: как «лучше» определять (бенчмарки/цена/свежесть), формат предложения смены, подтверждение в 1 тап.

5. **Не мешать всё в кучу.** Явно раздели: (default=чат) отдельно, каждая функция отдельно, служебные (squeeze/vision) отдельно. Предложи чистую структуру команд/меню, чтобы пользователь всегда понимал, какая модель где работает (у нас уже есть бейдж модели под ответом — учти это).

Формат ответа: структурировано, с таблицами и конкретными рекомендациями моделей по именам из каталога. Без воды.
