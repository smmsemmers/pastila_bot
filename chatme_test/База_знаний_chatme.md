# База знаний: платформа chatme.ai (для Support Manager)

Составлено по официальным докам chatme.ai (docs.chatme.ai) + движок GMS (developers.gms.net). Актуально для тестового Лены.

## 1. Что такое платформа и как устроена
chatme.ai — система создания/эксплуатации/управления чат-ботами. Умеет: NLU-сценарии (распознавание естественного языка), линейные сценарии (опрос/интервью/сбор данных), автоматизацию (перевод на группы операторов по условию), многопользовательский режим, вывод в каналы, интеграции с внешними системами (CRM и пр.), аналитику.

### Иерархия объектов
- **Платформа** → **Аккаунт пользователя** (по e-mail) → **Компания** (рабочее пространство) → **Проект/Папка** (группировка) → **Агент** (конфигурация бота = сценарий + настройки) → **Бот** (агент, выведенный в канал).
- **Собеседник** — с кем общается бот. **Чат** — контейнер всего общения собеседника с агентом в канале. **Диалог** — хронологический отрезок общения внутри чата.
- **Ресурсы Компании/Проекта** — общие: **Интенты**, **Внешние запросы**, **Словари** (Vocabularies) — доступны всем агентам.
- **Роли**: Owner (владелец), Administrator, и др. приглашённые. (Проверяющего теста приглашаем как **Administrator**.)

### Вкладки платформы (верхняя панель)
**Dashboard** (проекты/агенты) · **Resources** (Intents, External Requests, Vocabularies) · **Analytics** · **Users** (управление пользователями).
Внутри агента (Design): **AGENT DESIGNER** · **INTENTS** · **VOCABULARIES** · **EXTERNAL REQUESTS**. Справа вверху — **TRAIN** и тумблер **DEBUG**.

## 2. Сценарий и слоты
**Сценарий агента** строится в **БотБилдере (Agent Designer)** из **Слотов** — визуальных шагов-действий. Разворачивается слева направо от корневого **Start (ROOT)**.
- **Самостоятельные слоты** и **Комплексные** (Родитель + Подслоты/дочерние — дают ветвление).
- **Ветка сценария** — участок после разветвления родителя или после Start. **Родитель ветки** — слот, с которого ветка начинается.
- **Черновик сценария** — то, что редактируем; **Обученная модель** — скомпилированная рабочая версия после TRAIN.

### Типы слотов (полный перечень)
**Независимые:**
- **Start** — корень сценария (ROOT).
- **Text** — текстовый ответ бота (поле *Text List*). Тумблеры *Savepoint*, *Markdown*. Может содержать переменные `{{ var }}`.
- **Button Menu** (+ подслоты **button**, **fallback**) — кнопочное меню. У button: *Label* = текст кнопки (виден собеседнику), *Name* = имя в дереве. Если очистить Name — в дереве отображается Label. **fallback** — ветка, если собеседник ввёл текст, не совпадающий ни с одной кнопкой. ⚠️ Если введённый текст СОВПАДАЕТ с Label одной из кнопок — идём по её ветке (регистр игнорируется).
- **Slot Filling / Filling** — собрать ответ(ы) собеседника в переменную. Поле *Question* (вопрос) + *Answer variable* (имя пользовательской переменной для ответа). Неверное имя переменной → Validation error.
- **Memory** — создать/записать **пользовательскую контекстную переменную**. Поля: *Name* (имя слота) + пары **Key–Value** (Add new field). Value поддерживает выражения `{{ }}`, конкатенацию строк в одинарных кавычках. ⭐ Главный слот для создания переменных.
- **External Request** — HTTP-запрос во внешнюю систему (см. §5).
- **Change Chat Mode** — перевод диалога на оператора (в омниканальных каналах).
- **Incoming Request** — приём внешнего HTTP-запроса на вебхук агента → запускает ветку (рассылки/уведомления, бот пишет первым).
- **Notification** — рассылочные сообщения в WhatsApp (когда чата ещё нет / нельзя писать первым).
- **Timer** — запуск ветки через заданный период.
- **Attachment** — отправка файлов; поддерживает переменные.
- **Jump** — переход на произвольный слot (поле *Destination* — поиск слота по номеру/названию). Каждый слот в дереве имеет свой номер (ID).
- **Wait for Reaction** — блокирующий слот: агент встаёт в ожидание входящего сообщения собеседника. Ключевой для «зацикливания» диалога.

**Комплексные (с подслотами):**
- **NLU** (+ **intent** + **fallback**) — распознавание намерений. intent-подслоты привязаны к интентам; fallback — если ничего не распознано (ниже Threshold).
- **Transition Rule** (+ **condition**) — ветвление по условию. Автосоздаются подслоты: `main` и `FALLBACK_CONDITION`. В condition — поле *Condition* с логическим выражением (напр. `{{ raw_response == '[]' }}`); при `true` идём в его ветку.
- **Regular Expression** (+ **result**) — проверка/захват формата введённых данных по регулярному выражению.

**Правила соседства слотов** — какие слоты можно ставить рядом; при нарушении будет ошибка валидации.

## 3. Интенты и обучение (NLU)
- **Интент** — намерение собеседника в реплике, которое бот должен распознать.
- **Обучающая выборка** — набор фраз одного смысла, на которых обучается NLU распознавать интент. Заводится на вкладке **INTENTS**.
- **Threshold (Confidence)** — минимальная уверенность (0–100%), при которой интент считается распознанным и общение уходит в ветку его intent-подслота. Ниже порога → fallback.
- **Обучение (TRAIN)** — компиляция черновика в рабочую модель + **валидация сценария** (проверка требований системы). Ошибки подсвечиваются в дереве (см. статью «Требования для успешного обучения Агента · Ошибки валидации и их обозначение в Дереве»). Исправил → снова TRAIN.
- Можно **дообучать** интенты новыми фразами прямо со страницы Аналитики.

## 4. Контекстные переменные
- **Контекстная переменная** — хранится в чате, доступна для чтения/записи по логике сценария.
- **Пользовательские** (создаёт диалог-дизайнер: слоты **Memory**, **Filling**) vs **Системные** (создаёт платформа) vs **Канальные** (приходят из канала с каждым сообщением: логин, id чата и пр., напр. `client_message` = текст сообщения собеседника).
- **Контекст Чата** — весь набор переменных чата.
- **Где посмотреть значения по конкретному чату:** 1) в **DEBUG** → вкладка **Context variables**; 2) в **Аналитике** → «Выгрузка контекстных переменных чатов».

## 5. External Request (внешние запросы) — детально
Заводится на **Resources → External requests → Create new**. Вкладки: **Main, Headers, Query Parameters, Data, Response, Test**.
- **Main**: Name, Description, HTTP-метод (GET/POST/PUT/PATCH/DELETE/HEAD), **Endpoint** (URL, до 1000 симв., поддерживает `{{ var }}`).
- **Headers**: поддерживают `{{ var }}` и конкатенацию.
- **Response**: парсинг ответа — Data type (JSON/XML/TEXT), пары *Variable* → *Value*. Обычно `Variable = result`, `Value = {{ body }}` — всё тело ответа сохраняется в переменную `result`.
- В сценарии добавляется слот **External Request**, в поле *Request* выбирается созданный запрос.

**Служебные переменные после выполнения:**
- **`response_status_code`** — HTTP-код (400 — ошибка выполнения; 408 — таймаут 30 сек; 499 — нет ответа сервера).
- **`request_success`** — boolean (успех по кодам и валидности тела).
- **`raw_response`** — JSON `{"success":true/false}` + тело/текст ошибки. Если тело превышает лимит — ответ пишется сюда. Пустой ответ выглядит как `[]`.

## 6. Синтаксис выражений `{{ }}` (важно!)
- Подстановка переменной: `{{ client_message }}`, `{{ url }}`.
- Обращение к элементу массива (нумерация с 0): `{{ result[0].lat }}`, `{{ result[0].lon }}`.
- Обращение к элементу словаря: `result2["main"]`.
- Конкатенация строки: значение в одинарных кавычках с вставками, напр.
  `'http://api.openweathermap.org/geo/1.0/direct?q={{ client_message }}&appid=API_KEY'`.
- Сравнение (в Condition): `{{ raw_response == '[]' }}`.
- Есть **выражения с управляющими конструкциями** (условия/циклы) — раздел «Синтаксис».

## 7. Каналы и вывод в Telegram
Каналы: Telegram, WhatsApp (360dialog/cloud), Jivo, Webim, VK, Viber, Bitrix24, Chat2Desk, Microsoft Teams, Line, Widget, Omnidesk, Edna и др. + универсальный **Chat API**.
**Telegram:** карточка агента → **Channels → Telegram → Messaging** → вставить **token** от @BotFather → SAVE. Затем `/start` боту в мессенджере.

## 8. Аналитика (вкладка Analytics)
- Выгрузка отчёта по использованию агента (попадания в слоты).
- Выгрузка истории диалогов.
- **Выгрузка контекстных переменных чатов** (значения переменных по чатам).
- Выгрузка статистики прохождения по слотам.
- Отчёт по слотам завершения диалога.
- Фильтры: каналы, агенты, период, Dialog ID, Channel visitor ID. Таблицы: Chats / Dialogs / Requests; колонки Status (recognized), Intent, Confidence.

## 9. Настройки агента
- **Таймаут неактивности** — по умолчанию **15 мин**; по истечении без событий диалог закрывается.
- **Сбрасывать состояние общения** — начинать новый диалог с начала сценария, а не с места остановки.
- Экспорт/импорт/клонирование/замена сценария (файл `.cfg`), напр. `Проект-Опрос_сотрудников-2024-01-18-11-05.cfg`.

## 10. Ответы на 3 вопроса теста (обоснованы докой)
1. **Телефон 8→+7:** да. Собрать номер (**Filling**/переменная `client_message`), проверить формат слотом **Regular Expression**, затем слотом **Memory** переопределить значение через выражение-конкатенацию (как строится `url` в примере с погодой) — заменить ведущую «8» на «+7». Итог — нормализованный номер в переменной.
2. **Пользовательская переменная:** создаётся слотом **Memory** (Key–Value) или **Filling** (Answer variable). Значения по конкретному чату — в **DEBUG → Context variables** либо в Аналитике «Выгрузка контекстных переменных чатов».
3. **External Request не вернул заказы:** воспроизвести на конкретном чате → проверить `response_status_code` / `request_success` / `raw_response` → проверить формат номера (8 vs +7 vs формат в БД заказчика — частая причина, `[]` = пусто) → проверить парсинг Response (путь к данным) → проверить порядок слотов (переменная телефона заполнена до запроса) → дёрнуть API заказчика напрямую (Postman) с тем же номером → вывод: баг формата/маппинга — правим и ретест; нет заказов в БД — не баг бота.
